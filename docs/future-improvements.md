# Future Improvements

## Schema-Driven Validation (High Priority)

### Current Problem
The validation script (`R/validate_catalog.R`) has extensive hard-coded validation logic:
- Hard-coded regex patterns for UID and canonical filenames
- Hard-coded enum lists (doc types, languages, temporal types, subcategories, etc.)
- Duplicated validation logic between schema definition and validation code
- Risk of schema and validator getting out of sync

**Example of the problem:**
```r
# In validate_catalog.R (hard-coded):
valid_doc_type <- c("master", "share", "pumf")
valid_subcategories <- c("main", "simp", "subs", "freq", "rev", "alt", "comp", "synt", "spec")
uid_pattern <- "^cchs-[0-9]{4}[sdm]-[msp]-[a-z-]+(-[a-z]{4})?-(e|f)-(pdf|...)-[0-9]{2}$"

# In cchs_schema_linkml.yaml (also defined):
DocTypeEnum:
  permissible_values:
    master: ...
    share: ...
    pumf: ...
```

This duplication led to issues in v3.0.0 development where:
- Schema was updated but validator wasn't updated correctly
- UID pattern parsing logic was complex and error-prone
- No automated check that validator matches schema

### Proposed Solution: LinkML-Generated Validation

Use LinkML's built-in validation capabilities instead of custom R code:

```r
# Future approach:
library(linkml)  # hypothetical R LinkML bindings
library(yaml)

# Load schema
schema <- linkml::load_schema("metadata/cchs_schema_linkml.yaml")

# Validate catalog directly against schema
catalog <- yaml::read_yaml("data/catalog/cchs_catalog.yaml")
validation_results <- linkml::validate(catalog, schema)

# All validation rules come from schema:
# - Enum membership
# - Pattern matching
# - Required fields
# - Type checking
```

### Benefits

1. **Single Source of Truth**: Schema definition IS the validation
2. **Automatic Updates**: Change schema → validation automatically updated
3. **Better Error Messages**: LinkML knows context and can give precise errors
4. **Cross-Language**: Same schema can validate in Python, R, or other languages
5. **Standard Compliance**: Follows LinkML/JSON-Schema best practices

### Implementation Plan (Future Version)

**Phase 1: Research** (v3.1 or v4.0)
- Investigate LinkML Python validation tools
- Check if R bindings exist or if we need to call Python from R
- Prototype validation with small catalog subset

**Phase 2: Hybrid Approach** (v3.1)

- Add automated tests that compare validator against schema
- Generate validator code from schema (code generation)

**Phase 3: Full Migration** (v4.0)
- Replace custom validator with schema-driven approach
- Add LinkML validation as CI/CD check
- Document migration path for existing catalogs

## YAML DDI Extraction Gaps (Critical - Requires Re-extraction)

### Issue discovered

The YAML data dictionary extractions (in `cchs-extracted/data-dictionary/`) are **incomplete**. Cross-validation revealed significant gaps:

| Cycle | YAML Extracted | PDF Variable Lines | XML DDI | Gap Analysis |
|-------|----------------|-------------------|---------|--------------|
| cchs2015_2016_p | 639 | 1,283 | 1,283 | **Missing ~50% of variables!** |
| cchs2007_2008_p | 1,820 | 1,924 | 1,195 | Missing 104 variables from PDF |
| cchs2009_2010_p | 1,254 | 1,254 | 1,254 | Complete |

Specific missing variables discovered:
- cchs2015_2016_p: SMKG040 (and ~644 other variables)
- cchs2007_2008_p: SMKG01C, SMKG203, SMKG207 not in PDF (they're PUMF-only grouped vars)

### Root cause analysis

Investigation revealed **two distinct issues**:

#### Issue 1: Extraction bug (Critical)

The current extraction script (`scripts/extract_data_dictionary.R`) works correctly now, but the YAML files in the repo were extracted with an incomplete version. Evidence:

- Running the current script on 2015-2016 PDF extracts **1,283 variables**
- The committed YAML file has only **639 variables**
- The script handles leading whitespace correctly (`^\\s*Variable Name:`)
- The PDF has two formatting patterns: some lines start at column 1, others have leading space
- Only the non-whitespace-prefixed lines (639) were captured during initial extraction

**Root cause**: The extraction script was committed alongside the extractions, but the extractions were generated by an earlier (buggy) version before the whitespace fix was added.

#### Issue 2: PDF vs XML source differences (Expected)

Some variables exist only in XML DDI files (PUMF-only grouped/derived variables like SMKG01C, SMKG203). This is expected behaviour - the PDF data dictionaries document different variable sets than the XML DDI:
- **PDF data dictionaries**: Document all variables in the release (Master or PUMF)
- **XML DDI files**: Only available for PUMF releases

### Immediate fix required

**Priority 1: Re-run extraction for all PUMF data dictionaries**

```bash
# Re-extract 2015-2016 (and verify count matches PDF)
Rscript scripts/extract_data_dictionary.R \
  cchs-pumf-docs/CCHS-PUMF/CCHS_data_dictionary/2015_2016PUMFDataDictionary.pdf \
  cchs-extracted/data-dictionary/2015-2016/cchs_2015d_dd_p_en_1_v1.yaml \
  --metadata year=2015-2016 temporal_type=dual doc_type=pumf language=EN
```

**Priority 2: Add extraction validation check**

After each extraction, validate variable count matches PDF source:

```r
# Add to extraction workflow
pdf_var_count <- system2("pdftotext", c("-layout", pdf_file, "-"), stdout = TRUE) |>
  paste(collapse = "\n") |>
  strsplit("\n") |>
  unlist() |>
  grep("Variable Name:", x = _, value = TRUE) |>
  length()

yaml_var_count <- length(extracted_variables)

if (yaml_var_count != pdf_var_count) {
  warning(sprintf("Extraction incomplete: YAML has %d vars, PDF has %d",
                  yaml_var_count, pdf_var_count))
}
```

### Why cross-validation approach failed

The original workflow had several gaps:

1. **No ground truth comparison**: Extraction counted "success" if YAML was generated, not if variable count matched source
2. **No extraction-time validation**: Script version 1.1.0 logged `variables_count: 639` without comparing to PDF source
3. **Trusted process over verification**: Assumed PDF extraction was complete without cross-checking
4. **XML DDI used for wrong purpose**: XML DDI was checked for missing variables, but it has different variable set than PDF

### Proposed validation improvements

**Tier 1: Extraction-time validation**
- Compare extracted count vs PDF `Variable Name:` line count
- Fail extraction if counts don't match (with tolerance for known edge cases)
- Log both counts in extraction_summary.yaml

**Tier 2: Cross-source validation**
- For PUMF files, also compare against XML DDI variable names
- Flag variables in XML but not in YAML (expected: grouped/derived vars)
- Flag variables in YAML but not in XML (could indicate parsing artifacts)

**Tier 3: Semantic validation**
- Verify position values are sequential and non-overlapping
- Verify variable names match expected patterns (e.g., module prefixes)
- Cross-reference with questionnaire to verify question text alignment

### Impact on cchsflow

The `R/source-lookups.R` validation functions in cchsflow use the YAML DDI files. Current workaround:

1. When variable lookup fails, check XML DDI as fallback
2. If found in XML but not YAML, likely a grouped/derived PUMF variable
3. If not found in either, the variable genuinely doesn't exist in that cycle
