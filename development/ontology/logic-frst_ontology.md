# **Architectural Blueprint: Implementing a Logic Layer for cchsflow**

## **1\. Executive Summary**

The cchsflow framework currently operates on an **Instruction-Based** architecture. The variable\_details.csv file functions as a linear script, telling the software *how* to process a variable (e.g., recEnd: Func::convert\_inches) but failing to capture *why* that processing is correct or how variables relate across "multiplicative eras" of survey changes.  
As the number of CCHS cycles increases, this flat CSV model has become brittle. It forces the duplication of logic across rows and obscures the complex lineage of "interwoven questions" (e.g., variables that split, merge, or change universe over time).  
This report assesses the proposal to implement an **Ontology Layer** using **YAML**.

* **Recommendation:** Adopt **YAML** as the authoritative "Source of Truth" for variable definitions.  
* **Architecture:** Use a **Compiler Pattern**. The rich, nested YAML file (The Logic) is used to auto-generate the flat variable\_details.csv (The Instruction/View).  
* **Benefit:** This satisfies the requirement for a "lingua franca" (CSV) for analysts while providing the structural hierarchy (YAML) required to model complex longitudinal lineage.

## ---

**2\. The Architectural Challenge: The Limits of CSV**

You asked whether to add to the existing CSVs or create a new infrastructure. The challenge you face—"multiplicative eras" and "interwoven questions"—is fundamentally a **topology problem**.

* **The CSV Limit:** CSVs are 2-dimensional (Rows ![][image1] Columns). They excel at representing **Lists** (e.g., a list of instructions). They are poor at representing **Trees** or **Graphs** (e.g., a variable derived from two parent variables, which are in turn derived from a concept).  
* **The "Eras" Problem:** To represent a variable like BMI that changes measurement units in 2005, a CSV requires you to repeat the metadata for every cycle in "Era A" (2001, 2003\) and every cycle in "Era B" (2005–2018). If you find a bug in the "Era A" logic, you must manually update multiple rows.  
* **The Solution:** YAML (or JSON) supports **Nesting**. You can define "Era A" once as a block and simply list the cycles that belong to it.

## ---

**3\. The Proposed Solution: The "Compiler" Pattern**

We recommend a **Source vs. Artifact** architecture.

1. **The Source (YAML):** The cchs\_ontology.yaml file. This contains the "Logic," "Evidence," and "Relationships." It is edited by the package maintainers.  
2. **The Compiler (R Script):** A function (e.g., compile\_variable\_details()) that reads the YAML and flattens it.  
3. **The Artifact (CSV):** The variable\_details.csv file. This is generated by the compiler. It is read by cchsflow for execution and by analysts for review.

### **3.1 YAML Schema Design**

The schema below implements the **DDI-CDI Variable Cascade** 1, mapping your requirements (R1–R3) to a concrete structure.

#### **Example 1: Handling "Eras" of Change (Unit Drift)**

*Scenario: Height was measured in inches (2001–2003) and meters (2005+).*

YAML

\# concept: The abstract idea (Invariant)  
\- concept\_id: "HWTGBMI"  
  label: "Body Mass Index"  
  description: "BMI calculated from self-reported height and weight."  
  domain: "Physical Health"  
    
  \# representations: The "Eras" of measurement logic  
  representations:  
      
    \# Era 1: Imperial Units  
    \- era\_id: "BMI\_Imperial"  
      definition: "Calculated using height in inches and weight in lbs."  
        
      \# LOGIC LAYER (R1.2, R1.3)  
      \# Documentation of WHY and HOW  
      harmonization:  
        status: "partial"  
        rule\_category: "unit\_conversion"  
        evidence: "Source uses inches; target requires meters. Precision loss expected."  
        instruction: "Func::convert\_inches\_meters" 

      \# INSTANCE LAYER (R1.1)  
      \# The cycles belonging to this Era  
      instances:  
        \- { cycle: "cchs2001\_p", variable: "HWTAGBMI" }  
        \- { cycle: "cchs2003\_p", variable: "HWTAGBMI" }

    \# Era 2: Metric Units  
    \- era\_id: "BMI\_Metric"  
      definition: "Calculated using height in meters and weight in kg."  
        
      harmonization:  
        status: "complete"  
        rule\_category: "direct\_mapping"  
        instruction: "copy"

      instances:  
        \- { cycle: "cchs2005\_p", variable: "HWTCGBMI" }  
        \- { cycle: "cchs2007\_p", variable: "HWTDGBMI" }  
        \#... list continues for all metric years...

#### **Example 2: Interwoven Questions (Branching Logic)**

*Scenario: "Smoking Status" is a single question in early cycles but derived from two questions in later cycles.*

YAML

\- concept\_id: "SMK\_01"  
  label: "Type of Smoker"  
    
  representations:  
    \# Era 1: Direct Question  
    \- era\_id: "SMK\_Direct"  
      harmonization:  
        rule\_category: "recode"  
        instruction: "1::1; 2::0"  
      instances:  
        \- { cycle: "cchs2001\_p", variable: "SMK\_202" }

    \# Era 2: Derived from Frequency \+ Current Status  
    \- era\_id: "SMK\_Derived"  
      harmonization:  
        rule\_category: "case\_when"  
        evidence: "In 2005, 'Type of Smoker' was split into 'Frequency' and 'Current Status'."  
        instruction: "Func::merge\_smk\_vars"  
        
      \# Defining relationships between variables (R2.2)  
      input\_dependencies:  
        \- "SMK\_FREQ"  
        \- "SMK\_CURR"  
      instances:  
        \- { cycle: "cchs2005\_p", variable: "DerivedVar::SMK\_01" }

## ---

**4\. Mapping to Requirements**

This YAML structure directly satisfies the requirements drafted in your request.

| ID | Requirement | How YAML Solves It |
| :---- | :---- | :---- |
| **R1.1** | **Store relationships** | The instances list explicitly links source variables to the parent concept\_id and era\_id. |
| **R1.2** | **Classify relationships** | The harmonization.rule\_category field (e.g., direct\_mapping, recode) classifies the link type.3 |
| **R1.3** | **Document evidence** | The evidence field allows free-text rationale, decoupled from the executable code. |
| **R3.2** | **Group under abstract concepts** | The top-level YAML keys (concept\_id) provide the abstraction layer missing in the CSV. |
| **NF1** | **Human-readable/editable** | YAML is text-based and supports comments, making it easier for domain experts to review "Eras" rather than 1,000+ CSV rows. |
| **NF2** | **Version controllable** | Changes to logic are localized. Changing the BMI\_Imperial rule involves editing 1 block, not 20 rows. |
| **NF3** | **Queryable** | The YAML can be loaded into R (yaml::read\_yaml) as a list of lists, making it fully queryable for discovery tools. |

## ---

**5\. Implementation Strategy**

You do not need to rewrite the core rec\_with\_table function. You can implement this as a pre-processing step.

1. **Create inst/extdata/cchs\_ontology.yaml:** This becomes the master record.  
2. **Create data-raw/compile\_metadata.R:** This script reads the YAML and generates the CSV.  
   * *Input:* cchs\_ontology.yaml  
   * *Logic:* Loop through Concepts ![][image2] Eras ![][image2] Instances.  
   * *Output:* variable\_details.csv (The exact format you have today, perhaps with extra "Documentation" columns).  
3. **Workflow:**  
   * Developer adds a new variable to the YAML.  
   * Developer runs compile\_metadata.R.  
   * Developer commits both the YAML (source) and CSV (artifact).  
   * Users/Analysts can still open the CSV in Excel to see the "View" of the data.

### **5.1 The "CSV as a View"**

By treating the CSV as a generated artifact, you preserve its value as a "lingua franca." Analysts who want to see "What is the rule for 2005?" can simply filter the CSV. They get the **simplicity of the table** without the **complexity of maintaining it**.

## **6\. Conclusion**

Moving to a **YAML-based Ontology Layer** is the correct architectural decision for the scale cchsflow has reached. It solves the "Multiplicative Eras" problem by normalizing the logic into "Representations" (Eras), significantly reducing redundancy and error. By keeping the CSV as a generated output, you maintain compatibility with existing workflows and tools while gaining the power of a graph-like data structure for management.
